\# SecureERP — Master Context\## PropósitoEste documento es la fuente de verdad principal del proyecto SecureERP. Resume el contexto general, objetivos, decisiones base y restricciones obligatorias.\## Visión del proyectoSecureERP es un ERP enterprise moderno, seguro, multi-tenant, multiempresa, orientado a operaciones empresariales con seguridad tipo banca, aunque la organización no sea banco.\## Objetivos principales\- construir una plataforma ERP segura, modular y escalable\- soportar múltiples tenants, empresas, sucursales y unidades organizativas\- implementar IAM enterprise con RBAC + ABAC\- implementar RLS y aislamiento de datos por tenant/empresa\- habilitar navegación dinámica por permisos\- habilitar workflow de aprobaciones\- habilitar bloques transversales de documentos, comentarios y etiquetas\- garantizar logging técnico, auditoría funcional y eventos de seguridad desde día 0\## Stack principal\- .NET 8\- ASP.NET Core Web API\- WinForms\- DevExpress v25\- SQL Server\- Stored Procedures endurecidos\- arquitectura modular enterprise\## Principios obligatorios1\. Zero Trust2\. Least Privilege3\. Deny by default4\. Async en backend5\. Seguridad por capas6\. Logging desde día 07\. Auditoría obligatoria8\. Mensajes humanos para usuario y técnicos para dev9\. UI dinámica basada en recursos y acciones10\. documentación obligatoria y actualizada\## Baseline funcional\- multi-tenant\- multiempresa\- multisucursal\- unidades organizativas\- usuarios, roles, permisos, excepciones y scopes\- workflow de aprobaciones\- documentos\- comentarios\- etiquetas\- navegación dinámica\- auditoría y cumplimiento\## Reglas duras\- no rediseñar radicalmente el modelo\- no crear policy RLS paralela\- no crear motores alternos a las funciones canónicas\- no cambiar ALLOW y DENY\- no dejar seguridad en frontend\- no exponer errores técnicos al usuario final\- no hacer CRUD trivial sin controles enterprise\## Autoridad del documentoTodo desarrollo nuevo debe alinearse con este documento y con los baselines específicos relacionados.\---Autor: Mario R.
